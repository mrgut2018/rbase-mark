# 分类器代码逻辑

## 整体架构

分类系统采用**策略模式**，通过 `ClassifyAgent` 统一入口，根据分类器类型和方法动态选择具体实现类。

```
ClassifyAgent.classify()                    # 统一入口
  ├── 缓存检查（内存字典，按 classifier.alias 缓存）
  ├── 前置条件检查（prerequisite）
  ├── get_classifier_agent_impl()           # 策略选择
  │     ├── GeneralValueClassifier          # 通用值分类
  │     ├── NamedEntityMatchingClassifier    # 命名实体匹配
  │     ├── NamedEntityRecheckClassifier     # 命名实体复核
  │     └── RouteClassifier                 # 路由分类（未实现）
  └── 累计 token 使用量
```

---

## 核心文件

| 文件 | 作用 |
|------|------|
| `deepsearcher/agent/classify_agent.py` | 分类器核心逻辑，包含所有分类器实现类 |
| `deepsearcher/agent/prompts/classify_prompts.py` | 分类 prompt 模板 |
| `scripts/classify_raw_article.py` | 分类执行脚本，控制调用流程和结果处理 |
| `deepsearcher/rbase/ai_models.py` | 数据模型：Classifier, ClassifierValue 等 |

## 入口流程：ClassifyAgent.classify()

**文件**: `classify_agent.py` 第 100-130 行

```
classify(classifier_id, raw_article, **kwargs)
  │
  ├─ 1. 加载 Classifier 对象
  ├─ 2. 缓存检查：result_cache[classifier.alias] 命中则直接返回（不消耗 token）
  ├─ 3. 前置条件检查：check_classifier_prerequisite()
  │     └─ 不满足则抛出 ClassifierPrerequisiteNotMetError
  ├─ 4. 选择实现类：get_classifier_agent_impl()
  ├─ 5. 注入参数：file_loader、rbase_oss_settings
  ├─ 6. 执行分类：impl.classify(raw_article, **kwargs)
  ├─ 7. 检查 LLM finish_reason（stop/length/content_filter/timeout）
  ├─ 8. 累计 token 使用量
  └─ 9. 返回 JSON 字符串
```

## 策略选择：get_classifier_agent_impl()

**文件**: `classify_agent.py` 第 179-194 行

```
get_classifier_agent_impl(classifier, **kwargs)
  │
  ├─ VALUE_CLASSIFIER 类型
  │    ├─ GENERAL_CLASSIFICATION  → GeneralValueClassifier
  │    └─ NAMED_ENTITY_MATCHING
  │         ├─ entity_recheck=False → NamedEntityMatchingClassifier
  │         └─ entity_recheck=True  → NamedEntityRecheckClassifier
  │
  ├─ ROUTE_CLASSIFIER 类型       → RouteClassifier（未实现，仅占位）
  │
  └─ 其他类型                    → 抛出异常
```

所有实现类统一使用 `self.reasoning_llm`（配置为 `deepseek-reasoner`，流式模式）。

## 分类器实现类详解

### 1. GeneralValueClassifier — 通用值分类

**文件**: `classify_agent.py` 第 250-278 行

**用途**: 让 LLM 从预定义的候选值列表中选择分类结果。例如判断文章是"人类研究"还是"动物实验"。

**流程**:
1. 加载分类器的 `ClassifierValue` 候选值列表
2. 加载文章全文（如果分类器要求 `is_need_full_text`）
3. 拼接 `GENERAL_VALUE_CLASSIFIER_PROMPT`，包含：
   - 分类器定义：name、purpose、target、principle、criterion
   - 候选值表格：value_table
   - 文章信息：标题、期刊名、摘要、关键词、全文
   - 输出格式要求
4. 调用 LLM（deepseek-reasoner，流式模式 + 思维链日志）
5. `json_strip()` 清理返回内容（去除 markdown 代码块标记）
6. 返回 JSON 字符串

---

### 2. NamedEntityMatchingClassifier — 命名实体匹配

**文件**: `classify_agent.py` 第 335-380 行

**用途**: 从文章中识别并提取命名实体（如具体的疾病名、菌种名）。

**流程**（两条路径）:

#### 路径一：复用历史分类结果（不调 LLM）
```
classify_params 中配置了 use_classifier_results = "另一个分类器alias"
  → 查数据库获取该文章在那个分类器下的已有结果
  → 组装为 JSON 直接返回（token 消耗 = 0）
```

#### 路径二：调用 LLM 提取实体
```
没有可复用的历史结果
  → 加载文章全文（可选）
  → 拼接 NAMED_ENTITY_CLASSIFIER_PROMPT
  → 调用 LLM 提取实体
  → json_strip() 清理
  → _ensure_json_format() 兜底（如果返回不是合法 JSON，再调一次 LLM 转换）
```

---

### 3. NamedEntityRecheckClassifier — 命名实体复核

**文件**: `classify_agent.py` 第 492-653 行

**用途**: 对首次 NER 提取结果进行二次确认，确保实体与词表匹配的准确性。

**继承自**: `NamedEntityMatchingClassifier`

**入口分两条路径**:

#### 新模式：_classify_with_context（多候选比对）
```
输入: candidates[(ClassifierValue, score, is_exact), ...] + entity_name
  │
  ├─ 1. 为每个候选构建分类路径
  ├─ 2. 格式化为 Markdown 候选表格
  ├─ 3. 从全文中提取实体出现的上下文（前后各300字，最多5段）
  ├─ 4. 拼接 NAMED_ENTITY_RECHECK_WITH_CONTEXT_PROMPT
  ├─ 5. AI 判断标准：
  │     - 语义是否完全或极其接近同一概念
  │     - 语义相近但含义范围有差异的不返回
  │     - 向量相似度仅供参考，以上下文实际含义为准
  └─ 6. 返回确认正确的项，不匹配返回 []
```

#### 旧模式：_classify_legacy（单值验证）
```
输入: 单个 classifier_value + value_routes
  │
  ├─ 1. 加载全文，构建路径表格
  ├─ 2. 拼接 NAMED_ENTITY_RECHECK_CLASSIFIER_PROMPT
  ├─ 3. AI 判断标准：
  │     - 角色判断：只保留"主角"，排除"工具"和"背景"
  │     - 位置优先：标题/摘要/关键词 > 结论/结果 > 引言
  ├─ 4. 容错机制：LLM 调用失败（如 token 超限）→ 去掉全文再重试
  └─ 5. 重试仍失败返回 []，不阻塞流程
```

**Recheck 结果说明**: 只返回确认正确的匹配项，未返回的等于被排除。

## 完整 NER 分类流程（三段式）

```
第一段：提取
  NamedEntityMatchingClassifier.classify()
  → LLM 从文章中提取原始实体名（如 "E. coli"）

第二段：检索（上层代码 classify_raw_article.py 处理）
  → 拿实体名去词表/向量库搜索候选匹配项
  → 返回多个候选 [(ClassifierValue, score, is_exact), ...]

第三段：确认
  NamedEntityRecheckClassifier.classify()（entity_recheck=True）
  → 把候选项 + 文章上下文交给 LLM 二次确认
  → 返回最终确认的匹配结果
```

## 缓存机制

**类型**: 纯内存 Python 字典（`dict`）

```python
self.classifier_result_cache = {}
# 结构: { "classifier_alias": [result_item1, result_item2, ...] }
```

| 操作 | 说明 |
|------|------|
| 写入 | 分类结果处理时，按 `classifier.alias` 写入 |
| 读取 | `classify()` 入口处检查，命中则构造 `total_tokens=0` 的假响应返回 |
| 清除 | 换一篇文章时 `.clear()` 全部清空 |

**用途**: 同一篇文章的多个分类器之间存在依赖关系时，避免重复调用 LLM。

## 前置条件检查：check_classifier_prerequisite()

**文件**: `classify_agent.py` 第 196-243 行

分类器可定义 `prerequisite`（前置条件），是一组规则列表：

```json
[
  {
    "classifier_alias": "依赖的分类器别名",
    "value_in": ["允许的结果值列表"],
    "status_in": ["允许的状态列表"]
  }
]
```

**检查逻辑**:
1. 遍历每个分类器的前置规则
2. 如果依赖的分类器在当前批次中（会被一起处理），跳过该规则
3. 否则查数据库验证：该文章在依赖分类器下的结果/状态是否符合要求
4. 任一规则不满足 → 返回 `False` → 抛出 `ClassifierPrerequisiteNotMetError`

**示例**: 分类器 B 要求"分类器 A 的结果必须是 `human_study`"，只有 A 已完成且结果匹配时，B 才会执行。

## 结果处理：_process_classification_results()

**文件**: `classify_raw_article.py` 第 455-518 行

```
LLM 返回的 JSON 字符串
  │
  ├─ 1. safe_json_loads() 解析 JSON
  ├─ 2. 统一包装为列表
  ├─ 3. 遍历每条结果：
  │     ├─ 尝试写入缓存（供后续分类器复用）
  │     ├─ _parse_classify_result() 解析详情
  │     │    → label_item_key, classifier_value, location, entity_name 等
  │     └─ _save_classification_result() 保存到数据库
  └─ 4. 异常处理：解析失败记录日志并抛出
```

## LLM 调用方式

| 配置项 | 值 |
|--------|-----|
| 模型 | `deepseek-reasoner`（reasoning_llm） |
| 流式模式 | `stream: true` |
| 思考模式 | 支持，通过 `reasoning_content` 返回思维链 |
| 思维链日志 | `cot_log_filename` 参数，记录到文件供排查 |

**流式 vs 非流式**: 两种模式对调用方无差异，都返回完整的 `ChatResponse`。但思维链日志仅在流式模式下生效。

## 调用入口（scripts/classify_raw_article.py）

### 单独分类 vs 合并分类

```python
if sub_classifier_ids is None:
    # 单独分类：单个分类器处理单篇文章
    result = classify_agent.classify(classifier_id, raw_article, ...)
else:
    # 合并分类：多个分类器合并为一次调用
    result = classify_agent.merged_classify([classifier_id] + sub_classifier_ids, raw_article, ...)
```

### Recheck 调用入口

```python
# 旧版：单值验证
invoke_classify_agent_recheck(classifier_id, raw_article, classifier_values, ...)

# 新版：多候选比对
invoke_classify_agent_recheck_with_candidates(classifier_id, raw_article, candidates, entity_name, ...)
```

两者都通过 `entity_recheck=True` 参数切换到 `NamedEntityRecheckClassifier` 实现。
